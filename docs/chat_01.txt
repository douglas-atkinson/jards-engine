HTML5 Game Engine — Phase 0 Blueprint v1
1. Purpose of Phase 0

Phase 0 is not about building a full engine.

It is about:

choosing the first target game,
defining the smallest believable engine core,
setting architectural rules early,
setting up tooling and repository structure,
creating the first milestone path so development stays controlled.

The goal is to leave Phase 0 with a clean foundation and a very small but real development plan.

2. First Target Game Selection
Chosen target: Whac-A-Mole v1
Why this is the right first game

Whac-A-Mole gives us:

one main interaction loop,
one or more animated entities,
mouse input,
score tracking,
timing/spawn logic,
collision or click hit testing,
simple UI/HUD,
game states such as title, play, and game over,
optional sound effects.

That is enough to force real engine decisions without dragging us into huge complexity.

Initial design direction

Recommended view: top-down
Initial scope: one mole, one hole, one hammer/click interaction

Minimal gameplay loop
Mole appears from a hole.
Player clicks mole before it disappears.
Successful hit increases score.
Missed opportunity or bad click may optionally count against player.
Game runs for a fixed time or fixed number of rounds.
Why not side-scroller first

A side-scroller introduces:

platform movement expectations,
gravity questions,
camera implications,
more complicated collision behavior,
larger visual requirements.

All of that is cool later. None of it helps us prove the basic engine core right now.

3. MVP Engine Requirements

The engine MVP should only include what is needed to support Whac-A-Mole cleanly.

Core systems for MVP
3.1 Game Loop

Need:

fixed or semi-fixed update loop,
render loop,
delta time support,
pause/resume support later.

For now:

use requestAnimationFrame
track elapsed time
separate update(dt) from render(ctx)
3.2 Scene / State Management

Need:

a way to switch between screens or modes.

Initial scenes:

BootScene
TitleScene
GameScene
GameOverScene

This keeps us from hard-wiring everything into one giant file.

3.3 Input System

Need:

mouse position,
mouse click/down/up,
maybe keyboard later.

For Whac-A-Mole v1:

mouse click is enough
optional touch abstraction would be nice to leave room for later
3.4 Rendering Layer

Need:

canvas creation/management,
draw sprite/image,
draw shapes,
draw text,
optional simple layering

For MVP:

HTML5 Canvas 2D is the right choice
no WebGL yet
no premature shader rabbit hole
3.5 Asset Management

Need:

image loading,
audio loading later,
preload support.

For MVP:

simple loader for images
optional sound hooks but not necessarily first implementation
3.6 Entity Model

Need:

a consistent way to represent game objects

For MVP:

keep it lightweight
entities can have:
id
position
size
active flag
update()
render()

We do not need full ECS on day one.

3.7 Collision / Hit Testing

Need:

point-in-rectangle or point-in-circle hit testing
maybe bounding boxes

For Whac-A-Mole:

click hit detection on mole sprite or mole hitbox
3.8 Timing / Scheduling

Need:

timers for mole rise/hide/spawn
possibly cooldowns and round timers

This can begin as a very small timer utility.

3.9 HUD / UI Text

Need:

draw score
draw timer
draw messages

Simple canvas text is enough for v1.

3.10 Audio Hooks

Need eventually:

hit sound
miss sound
title/game over sound

For MVP:

design for it
implement after visual loop works
4. Architecture Direction
Initial recommendation

Use a traditional object-oriented game object architecture first, while designing in a way that does not block an ECS transition later.

That means:

scene manager
entity/game object base class
managers/utilities for input/assets/timing
components only where they clearly help
Why not full ECS immediately

Because ECS is interesting, powerful, and absolutely capable of becoming a beautiful overengineered cathedral before we even have one mole popping out of one hole.

A full ECS at the beginning risks:

spending weeks on framework philosophy,
inventing abstractions we do not yet understand through use,
building systems for problems the first game does not have.
Better approach

Phase 0 and early Phase 1:

use a clean object-based model
isolate behavior enough that future ECS refactoring is possible

Then after first or second game:

review pain points
decide whether to evolve toward hybrid ECS or full ECS
Sensible middle ground

We can start discussing ECS concepts now:

data vs behavior
systems updating groups of entities
components as reusable data bundles

But we should earn those abstractions with real usage.

6. Initial API Sketch

This is deliberately lightweight. We want something usable, not fancy.

Game bootstrap
const game = new Game({
  width: 800,
  height: 600,
  parent: "game-container",
  startingScene: new TitleScene(),
});

game.start();
Base Scene concept
class Scene {
  enter() {}
  exit() {}
  update(dt) {}
  render(renderer) {}
}
Scene manager
sceneManager.changeScene(new PlayScene());
Entity concept
class Entity {
  constructor(x, y, width, height) {
    this.x = x;
    this.y = y;
    this.width = width;
    this.height = height;
    this.active = true;
  }

  update(dt) {}
  render(renderer) {}
}
Clickable entity concept
class ClickableEntity extends Entity {
  containsPoint(x, y) {
    return (
      x >= this.x &&
      x <= this.x + this.width &&
      y >= this.y &&
      y <= this.y + this.height
    );
  }

  onClick() {}
}
Input manager idea
input.isMouseDown();
input.wasMouseClicked();
input.getMousePosition();
Timer utility
const timer = new Timer(1000, () => {
  mole.show();
});
Renderer idea
renderer.clear();
renderer.drawImage(image, x, y, width, height);
renderer.drawRect(x, y, width, height);
renderer.drawText("Score: 10", x, y);

This is enough to get moving without locking us into nonsense.

12. Risks We Want to Avoid

Phase 0 exists partly to dodge these traps:

Trap 1: Building an engine instead of a game

We need a game-driven engine, not an abstract framework floating in space.

Trap 2: Premature ECS religion

ECS should solve problems, not create them.

Trap 3: Toolchain overkill

We do not need six libraries, hot physics, particle middleware, and a philosophical debate on architecture before the mole exists.

Trap 4: Feature creep before proof of concept

No level editor, no plugin architecture, no networking, no save system, no multiplayer mole combat empire.

Not yet, anyway.

WebStorm put index.html under src, not under root

Ahh yep—good catch. That’s WebStorm being “helpful” in a way that’s… not ideal for how Vite expects things 😄

Let’s straighten this out cleanly so we don’t fight the tooling later.

🔧 What Vite Expects (Important)

Vite expects:

project-root/
├── index.html   ✅ (must be here)
├── src/
│   └── main.ts

👉 index.html should be in the project root, NOT inside src/

Why this matters:

Vite treats index.html as the entry point
It uses it to build dependency graphs
If it’s inside src/, weird things can happen later (especially with builds)
✅ Fix (Super Simple)

In WebStorm:

Drag index.html out of src/
Drop it into the project root

So it becomes:

jards-engine/
├── index.html   ← move here
├── src/
│   └── main.ts
🔧 Double-Check This Line

Open your index.html and make sure this still works:

<script type="module" src="/src/main.ts"></script>

That /src/main.ts path assumes index.html is in the root—so once you move it, this becomes correct.

🚀 Then Test Again

Run:

npm run dev

If everything is right:

page loads
no console errors
your canvas (or Vite demo) shows up

Vite’s mental model is:

“Serve the project like a smart static site, but treat index.html as code.”

That’s why it lives at the root—it’s not just a file, it’s part of the build pipeline.
